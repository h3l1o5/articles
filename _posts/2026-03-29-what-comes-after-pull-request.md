每個使用 coding agent 的團隊，現在都面對同一個矛盾。

工程師的生產力感覺前所未有的高。原型在幾小時內就能跑起來。Pull request 以超出預期的速度堆積。然而，真正抵達終端使用者手中的可信賴軟體，成長速度卻沒有跟上。

Cognition 用一句改自邱吉爾的話精準描述了這個現象：軟體工程領域裡，從未有如此多的程式碼，被如此多的人創造出來，卻只被交付給如此少數的人。

## 新的瓶頸

如今 coding agent 基本上已能正常運作，團隊生成程式碼的速度比以往都快。但撰寫程式碼只是軟體開發生命週期的一部分而已。我們在加速程式碼生成的同時，卻沒有同等加速 review、理解、驗證與信任的流程。

更廣泛的數據也指向同一個方向。Faros AI 調查了 1,200 多個團隊的超過 10,000 名開發者，發現高 AI 採用率的團隊完成的任務多 21%、合併的 pull request 多 98%，但每天花在 review 程式碼的時間也多了 91%。

**這，就是新的瓶頸。**

我們圍繞軟體交付所建立的整套工具鏈，是基於「程式碼由人以人的速度撰寫」這個假設。Pull request、review queue、CI、程式碼所有權、審批流程、協作規範——這一切都從那個假設中衍生而來。在 agent 為主的世界裡，這個假設已不再成立。

這就是為什麼 AI 輔助軟體開發中，最重要的問題不再只是程式碼生成，而是**生成之後發生什麼事**。

## 三種賭注

當 agent 能大量快速產出程式碼，限制因素就從「撰寫」變成了「評估」。人類每次只能吸收這麼多的變更，只能 review 這麼多的 diff，只能在腦子裡記住這麼多的 context。隨著程式碼生成加速，所有下游環節都開始承受壓力：

- Review queue 越來越長
- 訊噪比越來越差
- 越來越多時間花在閱讀生成的程式碼，卻無法真正理解它

主要的 AI 公司正在競相解決這個問題。以下是三種主要路線：

### 賭注一：讓 Review AI 原生化

程式碼 review 是最早出現的 agentic 產品類別之一，CodeRabbit、Greptile 等公司都在建立協助團隊 review AI 生成程式碼的系統。然而開發者普遍認為，讓 AI 像另一個貢獻者一樣在 PR 上留言，訊噪比並不理想。

Cognition 因此推出了 Devin Review。核心思路很直觀：如果人類無法以 agent 的速度消化 diff，就給他們一個 AI 工具來填補這個落差。

具體來說：
- 智慧 diff 組織，依邏輯關係將變更分組
- 複製與移動偵測，讓檔案重新命名不會被誤判為完整的刪除再重寫
- 帶有 codebase context 的 inline chat，讓你在 review 途中就能問「這個變更會影響什麼」

最有趣的部分是它如何閉合回路：agent 寫程式碼，review agent 評估它、執行檢查、找出問題，review agent 的輸出再回饋給 coding agent，循環直到結果乾淨為止。這個閉合循環行為已在各主要系統中出現：Devin Review 透過 auto-loop 實現，Cursor 透過 computer use 實現。

賭注：人類留在迴路中，但程式碼 review 必須進化。它不會消失，而是變得 AI 原生。

### 賭注二：砍掉 Review

Ankit Jain 在 Latent Space 提出了這個論點：程式碼 review 本來就已經在崩潰了，在 coding agent 出現之前就是。症狀很熟悉：PR 擱置好幾天、橡皮圖章式批准、reviewer 因為自己的 sprint 工作而草草瀏覽 500 行的 diff。

我們稱程式碼 review 為品質關卡，但團隊在 2012 到 2014 年 GitHub 讓 PR 成為常態之前，幾十年來都沒有逐行 review 就在出貨了。Agent 只是讓這個矛盾變得無從忽視。

Jain 的替代方案是把人類的檢查點移到上游：人類 review 規格、計劃和驗收標準，以分層信任取代單一的下游關卡——BDD 規格作為信任來源、確定性的護欄、權限範圍、對抗性驗證、可觀察性，以及快速 rollback。

在這個模型中，人類定義應該發生什麼，系統驗證是否發生了。程式碼變成一個生成的 artifact，而不是主要的 review 對象。

賭注：review 本來就已經大半是走過場了。Jain 的提案是把陣地移到抽象層的更上游。

### 賭注三：替換整個平台

OpenAI 正在打造一個 GitHub 競爭者。

立即的導火線或許是 GitHub 的服務中斷，但更廣泛的戰略邏輯更耐人尋味。一個從頭為 AI 原生工作流程設計的 repository 平台，不會把 agent 當成人類優先系統中的笨拙參與者，而是把它們當成第一等公民。

同樣的方向也體現在 Symphony——OpenAI 開源的 agent 驅動軟體工作協作層。Symphony 監視專案看板、為任務啟動 agent、管理實作、執行 CI、處理 review 流程，並將工作推進到 PR 完成。工程師管理工作與意圖，agent 處理中間大部分的執行。

賭注：工具鏈本身就是瓶頸。修補 GitHub 現有的 review 流程，對於 agent 優先的世界來說可能還不夠。你需要一個 agent 與人類原生協作的新平台。

## 三種賭注的共同點

這三種路線各有不同，但它們匯聚在同一個更廣泛的轉變上：

**人類的角色正在從「你把這個寫對了嗎？」和「我有沒有逐行檢查？」移向「我們在解決正確的問題嗎？」、「約束條件正確嗎？」以及「我們有沒有把意圖表達得夠清晰？」**

價值正在往上游移動，移向意圖和規格，遠離 diff。

這符合軟體業作為「不斷提升抽象層次」的產業的歷史脈絡。程式設計師必須再次調整自己的工作流程，就像編譯器出現時一樣。

## 工程師的工作正在改變

OpenAI 做了一個他們稱為 harness engineering 的實驗。他們打造了一個百萬行的生產應用程式，其中零行由人類撰寫。工程師仍然很重要，但他們的角色變了：他們設計環境、規定意圖，並建立讓系統可靠的回饋迴路。

他們 postmortem 中的關鍵教訓是：

> 當某個環節失敗，答案幾乎從來不是「更努力試」，而是識別缺少什麼能力，並讓它對 agent 來說是可辨識且可執行的。

這在實務上意味著什麼：

**架構執行移入系統之中**——與其依賴人類 reviewer 抓出結構性錯誤，團隊越來越多地將架構直接編碼進工具中：相依規則、命名慣例、檔案大小限制、結構測試、帶有自定義錯誤訊息的 linter（專門教 agent 如何修正違規）。

**文件成為執行環境的一部分**——只存在於 Slack 討論串或某人腦袋裡的知識，對系統來說是不可見的。在 agent 為主的世界裡，這是嚴重的弱點。有用的文件必須作為結構化、版本化的 artifact 存在於 repo 中。對系統不可見的知識，等同於不存在。

**工程師越來越透過意圖來運作**——隨著 agent 承擔更多程式碼生成，工程師花越來越少時間撰寫實作，越來越多時間透過 prompt、規格、計劃、review 標準、回饋迴路和環境設計來工作。程式碼是 agent 的輸出；工程師的輸出是讓程式碼可信賴的系統。

Cognition 來自客戶部署的數據支持這一點：只有 20% 的工程時間花在 coding，大部分都在規劃、review 和 scope 界定上。隨著工具越來越好，這個比例還會進一步改變。

## 搭鷹架，然後被吞噬

這三種賭注各有不同的範疇和野心，但它們都有同一個結構性的形狀——**暫時的**。

Noam Brown 給出了一個有用的例子：在推理模型出現之前，團隊建立了複雜的鷹架來誘導較弱系統產生類推理行為：多次呼叫的協作、思維鏈結構、router、專用 harness。然後更好的模型突然出現，讓大部分的結構不僅不必要，甚至有害。

現有的證據也不斷指向同一個方向：
- METR 的發現顯示 Claude Code 和 Codex 在時間水平測量上，並未顯著優於基本的預設 scaffold
- Scale AI 的 SWE-Atlas 顯示原生 scaffold 對前沿模型有幫助，但隨著模型變強，差距在縮小
- Boris Cherny 描述 Anthropic 的做法：「我們字面上造不出比這更簡單的東西了」

然而，harness 在短期內仍是最高槓桿的變數。Can Boluk 的 harness 結果讓這一點具體化：僅改變 edit tool 就讓 15 個 LLM 的表現提升了 5 到 14 個百分點。

這是當下這個時刻核心的矛盾：**你現在能建立的最有影響力的東西，幾乎肯定是暫時的。**

但這並不是說不該建。在 review harness 裡建構的學習，教你在 agent context 中什麼是好的 review。建立 spec 層的學習，教你意圖需要是什麼樣子才能被執行。建立協作系統的學習，教你人類的交接點究竟在哪裡。

鷹架可能是暫時的，但學習不是。你必須建立解決當下問題的肌肉，才能賺到理解下一步是什麼的洞察。

## 還缺什麼

**可觀察性仍然薄弱**——團隊被要求在他們仍無法清楚檢視的工作流程上，建立高信任度的系統。各家 lab 不斷談論 context engineering、prompt 品質和系統行為，卻仍然提供非常有限的工具，來讓人視覺化和理解 context window 內部究竟發生了什麼。

**規格還無法取代 review**——讓人類判斷完全移至上游的論點，有賴於規格精確到足以生成和驗證。正如 Kevlin Henney 所說：「以無歧義的細節描述一個程式的行為，和程式設計本身是同一回事。」這是 spec 驅動開發的陷阱：一旦規格精確到機器可以執行，它就開始看起來像一種程式語言。人類意圖與機器執行之間的抽象層，仍然缺少它的編譯器。

## 結語

Pull request 不是因為有人發明了更好的 review 介面就消亡的。它正在消亡，是因為其底層的假設已不再成立——由人類以人的速度撰寫程式碼、由具有完整 context 的人類 review、透過建立在直接作者身份上的社會信任批准。這一切都無法描述 agent 為主的世界中軟體是如何被製作出來的。

取而代之的，將是這裡描述的這些賭注的某種組合：AI 原生的 review、更強大的上游規格，以及更 agent 原生的平台。這一切現在都是必要的，而且大概全都是鷹架。

能夠順利度過這個過渡的團隊，將是那些在人類意圖和已出貨的軟體之間，建立儘可能薄的一層、把每一層都視為有到期日的賭注，並且知道何時該拆掉它——因為模型已經趕上了——的團隊。

這是新的技術。不是建立精密的系統，而是把用後即棄的系統建好。

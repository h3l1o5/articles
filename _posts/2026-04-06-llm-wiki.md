## LLM Wiki

一種利用 LLM 建立個人知識庫的模式。

這是一份概念文件，設計上就是讓你直接複製貼到你自己的 LLM Agent（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目標是傳達高層次的概念，而你的 agent 會與你協作，把具體細節建構出來。

## 核心概念

大多數人使用 LLM 處理文件的體驗看起來像 RAG：你上傳一堆檔案，LLM 在查詢時擷取相關片段，然後生成答案。這能用，但 LLM 每次都在從零開始重新發現知識。沒有任何累積。問一個需要綜合五份文件才能回答的細緻問題，LLM 每次都得重新找到並拼湊相關片段。什麼都沒有被建立起來。NotebookLM、ChatGPT 檔案上傳，以及大多數 RAG 系統都是這樣運作的。

這裡的概念不同。不是在查詢時才從原始文件中擷取，而是讓 LLM 逐步建立並維護一個持久的 wiki——一個結構化、互相連結的 markdown 檔案集合，位於你和原始資料之間。當你加入一個新來源時，LLM 不只是為了日後擷取而索引它。它會閱讀它、提取關鍵資訊，並整合進現有的 wiki——更新實體頁面、修訂主題摘要、標註新資料與舊觀點矛盾之處、強化或挑戰不斷演進的綜合分析。知識被編譯一次，然後持續更新，而不是每次查詢都重新推導。

這就是關鍵差異：wiki 是一個持久的、不斷累積的產物。交叉引用已經建好了。矛盾已經被標記出來了。綜合分析已經反映了你讀過的所有內容。每增加一個來源、每問一個問題，wiki 都會變得更豐富。

你永遠不用（或很少需要）自己寫 wiki——LLM 負責撰寫和維護所有內容。你負責的是蒐集資料來源、探索，以及問對的問題。LLM 負責所有苦工——摘要、交叉引用、歸檔，以及讓知識庫隨著時間真正發揮作用的各種記帳工作。實際操作上，我一邊開著 LLM agent，另一邊開著 Obsidian。LLM 根據我們的對話進行編輯，我即時瀏覽結果——點擊連結、查看圖譜視圖、閱讀更新後的頁面。Obsidian 是 IDE；LLM 是程式設計師；wiki 是程式碼庫。

這個模式可以應用在很多不同的情境。以下是一些例子：

- **個人**：追蹤你自己的目標、健康、心理、自我提升——歸檔日記、文章、podcast 筆記，隨著時間建立起一幅結構化的自我圖像。
- **研究**：花數週或數月深入鑽研一個主題——閱讀論文、文章、報告，逐步建構一個包含不斷演進論點的完整 wiki。
- **閱讀一本書**：逐章歸檔，為角色、主題、情節線索以及它們之間的連結建立頁面。讀完後你就擁有一個豐富的伴讀 wiki。想想那些粉絲 wiki，像是 Tolkien Gateway——數千個互相連結的頁面，涵蓋角色、地點、事件、語言，由志願者社群花費數年建構。你可以在閱讀時自己建立類似的東西，由 LLM 負責所有交叉引用和維護工作。
- **商業/團隊**：由 LLM 維護的內部 wiki，餵入 Slack 討論串、會議逐字稿、專案文件、客戶通話。可能讓人類參與審核更新。wiki 保持最新狀態，因為 LLM 負責了團隊中沒人想做的維護工作。
- **競爭分析、盡職調查、旅行規劃、課程筆記、興趣深入研究**——任何你隨時間累積知識並希望它有組織而非散落各處的場景。

## 架構

共有三層：

**原始來源**——你精心蒐集的來源文件。文章、論文、圖片、資料檔案。這些是不可變的——LLM 從中讀取但絕不修改。這是你的事實來源。

**Wiki**——一個由 LLM 生成的 markdown 檔案目錄。摘要、實體頁面、概念頁面、比較、概覽、綜合分析。LLM 完全擁有這一層。它建立頁面、在新來源到達時更新它們、維護交叉引用，並保持一切的一致性。你負責閱讀；LLM 負責撰寫。

**Schema**——一份文件（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告訴 LLM wiki 的結構方式、慣例是什麼，以及在匯入來源、回答問題或維護 wiki 時應遵循的工作流程。這是關鍵的設定檔——它讓 LLM 成為一個有紀律的 wiki 維護者，而不是一個通用聊天機器人。你和 LLM 會隨著時間共同演進這份文件，逐漸找出適合你領域的方式。

## 操作

**匯入（Ingest）。** 你把新來源放入原始資料集合中，然後告訴 LLM 去處理它。一個範例流程：LLM 閱讀來源、與你討論關鍵重點、在 wiki 中撰寫摘要頁面、更新索引、更新整個 wiki 中相關的實體和概念頁面，並在日誌中附加一筆記錄。單一來源可能會觸及 10-15 個 wiki 頁面。就個人而言，我偏好一次匯入一個來源並持續參與——我閱讀摘要、檢查更新，並引導 LLM 該強調什麼。但你也可以一次批量匯入多個來源，減少監督。關鍵在於你要發展出適合你風格的工作流程，並將其記錄在 schema 中供未來的 session 使用。

**查詢（Query）。** 你針對 wiki 提出問題。LLM 搜尋相關頁面、閱讀它們，並以引用來源的方式綜合出一個答案。答案可以根據問題採取不同形式——一個 markdown 頁面、一張比較表、一份投影片（Marp）、一張圖表（matplotlib）、一個畫布。重要的洞見是：好的答案可以作為新頁面歸檔回 wiki。你要求的比較、分析、你發現的連結——這些都很有價值，不應該消失在聊天記錄中。這樣你的探索就和匯入的來源一樣，在知識庫中持續累積。

**檢查（Lint）。** 定期請 LLM 對 wiki 做健康檢查。檢查項目包括：頁面之間的矛盾、已被較新來源取代的過時觀點、沒有入站連結的孤立頁面、被提及但缺少獨立頁面的重要概念、缺失的交叉引用、可以透過網路搜尋填補的資料缺口。LLM 擅長建議新的研究問題和要尋找的新來源。這讓 wiki 在成長過程中保持健康。

## 索引與日誌

兩個特殊檔案幫助 LLM（和你）在 wiki 成長時進行導航。它們服務不同的目的：

**index.md** 是以內容為導向的。它是 wiki 中所有內容的目錄——每個頁面都列出連結、一行摘要，以及選擇性的 metadata 如日期或來源數量。按類別組織（實體、概念、來源等）。LLM 在每次匯入時都會更新它。回答查詢時，LLM 先讀索引找到相關頁面，再深入閱讀。這在中等規模（約 100 個來源、數百個頁面）下效果出奇地好，並避免了需要基於 embedding 的 RAG 基礎設施。

**log.md** 是按時間順序的。它是一個僅附加的記錄，記載了什麼事在什麼時候發生——匯入、查詢、檢查。一個有用的技巧：如果每筆記錄都以一致的前綴開頭（例如 `## [2026-04-02] ingest | Article Title`），日誌就可以用簡單的 unix 工具來解析——`grep "^## \[" log.md | tail -5` 就能給你最後 5 筆記錄。日誌讓你看到 wiki 演進的時間線，並幫助 LLM 了解最近做了什麼。

## 選用：CLI 工具

到了某個階段，你可能會想建立一些小工具，幫助 LLM 更有效率地操作 wiki。wiki 頁面的搜尋引擎是最明顯的一個——在小規模時索引檔案就夠用了，但隨著 wiki 成長，你需要正式的搜尋功能。qmd 是個不錯的選擇：它是一個本地端的 markdown 檔案搜尋引擎，具備混合 BM25/向量搜尋和 LLM 重新排序功能，全部在本機上運行。它同時有 CLI（讓 LLM 可以 shell out 呼叫）和 MCP server（讓 LLM 可以當作原生工具使用）。你也可以自己建一個更簡單的——LLM 可以在需要時幫你快速寫出一個樸素的搜尋腳本。

## 技巧與訣竅

- **Obsidian Web Clipper** 是一個瀏覽器擴充功能，可以將網頁文章轉換成 markdown。對於快速將來源納入你的原始資料集合非常有用。
- **在本地下載圖片。** 在 Obsidian 設定 → Files and links 中，將「Attachment folder path」設定為固定目錄（例如 `raw/assets/`）。然後在設定 → Hotkeys 中搜尋「Download」找到「Download attachments for current file」，並綁定一個快捷鍵（例如 `Ctrl+Shift+D`）。擷取文章後按下快捷鍵，所有圖片就會下載到本機。這是選用的但很有用——它讓 LLM 可以直接查看和引用圖片，而不是依賴可能失效的 URL。注意 LLM 無法在一次讀取中原生處理包含行內圖片的 markdown——變通方法是讓 LLM 先讀文字，再分別查看部分或全部引用的圖片以獲得額外脈絡。這有點笨拙但效果還不錯。
- Obsidian 的**圖譜視圖（graph view）** 是觀察你的 wiki 形狀的最佳方式——什麼跟什麼相連、哪些頁面是樞紐、哪些是孤立的。
- **Marp** 是一種基於 markdown 的投影片格式。Obsidian 有對應的 plugin。對於直接從 wiki 內容生成簡報很有用。
- **Dataview** 是 Obsidian 的一個 plugin，可以對頁面 frontmatter 執行查詢。如果你的 LLM 在 wiki 頁面加入 YAML frontmatter（標籤、日期、來源數量），Dataview 就能生成動態表格和清單。
- wiki 就是一個包含 markdown 檔案的 git repo。你免費獲得版本歷史、分支和協作功能。

## 為什麼這行得通

維護知識庫中最繁瑣的部分不是閱讀或思考——而是記帳工作。更新交叉引用、保持摘要最新、標註新資料與舊觀點的矛盾、在數十個頁面間維持一致性。人類會放棄 wiki，因為維護負擔的成長速度超過了它帶來的價值。LLM 不會感到無聊、不會忘記更新一個交叉引用，而且可以一次修改 15 個檔案。wiki 能持續被維護，因為維護成本幾乎為零。

人類的工作是策展來源、引導分析、提出好問題，並思考這一切的意義。LLM 的工作是其他所有事情。

這個概念在精神上與 Vannevar Bush 的 Memex（1945）有關——一個私人的、主動策展的知識儲存庫，文件之間有關聯性的路徑。Bush 的願景比後來的網路更接近這個概念：私密的、主動策展的，文件之間的連結與文件本身同等重要。他無法解決的部分是誰來做維護工作。LLM 解決了這個問題。

## 備註

這份文件刻意保持抽象。它描述的是概念，而非具體實作。確切的目錄結構、schema 慣例、頁面格式、工具——所有這些都取決於你的領域、你的偏好，以及你選擇的 LLM。上面提到的一切都是選用且模組化的——挑選有用的，忽略不需要的。例如：你的來源可能全是純文字，所以你完全不需要圖片處理。你的 wiki 可能小到索引檔案就夠了，不需要搜尋引擎。你可能不在乎投影片，只想要 markdown 頁面。你可能想要一套完全不同的輸出格式。正確的使用方式是把這份文件分享給你的 LLM agent，然後一起協作，實例化一個適合你需求的版本。這份文件唯一的職責就是傳達這個模式。你的 LLM 可以搞定剩下的。

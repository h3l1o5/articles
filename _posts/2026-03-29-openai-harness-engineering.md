> 原文：[Harness Engineering: Using Codex in an Agent-First World](https://openai.com/index/harness-engineering/) by **Ryan Lopopolo**
> 發佈日期：2026 年 2 月 11 日

在過去五個月中，我們的團隊一直在進行一項實驗：開發並發布一款內部 Beta 版的軟體產品，完全沒有手動撰寫的程式碼。

本產品有內部每日使用者和外部 alpha 測試人員。它會推出、部署、出錯，然後被修復。不同之處在於，每一行程式碼——應用程式邏輯、測試、CI 設定、文件、可觀測性，以及內部工具——都是由 Codex 撰寫的。我們估計，我們用大約手動撰寫程式碼所需時間的 1/10 就完成了這項工作。

人類掌舵，代理程式執行。

我們刻意選擇這項限制，以便我們能夠建造出必要的東西，從而使工程速度提升數個數量級。我們有幾週的時間來交付最終成為一百萬行程式碼的產品。為了達成這個目標，我們需要了解：當軟體工程團隊的主要任務不再是撰寫程式碼，而是設計環境、明確意圖，並建立回饋迴路以讓 Codex 智能代理能夠可靠地執行工作時，會發生什麼變化。

這篇文章談的是我們在與一個代理團隊合作開發全新產品時的學習經驗：哪些地方出現問題、哪些問題相互影響，以及如何最大化我們唯一真正稀缺的資源——人類的時間和注意力。

## 從空的 Git 程式碼庫開始

第一次提交到空的程式碼庫是在 2025 年 8 月下旬完成的。

初始架構，包括程式碼庫結構、CI 設定、格式化規則、套件管理器設定和應用程式框架，是由 Codex CLI 使用 GPT‑5 生成，並由一小組現有範本指導。甚至最初用來指導代理在程式碼庫中工作的 AGENTS.md 檔案，也是由 Codex 撰寫的。

沒有任何現有的人類撰寫程式碼可作為系統的基礎。從一開始，程式碼庫就是由智慧體塑造的。

五個月後，程式碼庫大約包含一百萬行程式碼，涵蓋應用程式邏輯、基礎設施、工具、文件和內部開發人員工具。在那段期間，大約有 1,500 個 pull request 被開啟並合併，而推動 Codex 的小型團隊僅有三位工程師。這相當於每位工程師每天平均吞吐量為 3.5 個 PR，而且令人驚訝的是，隨著團隊成長到如今七位工程師，吞吐量反而增加了。重要的是，這並非為了輸出而輸出：該產品已在內部被數百位用戶使用，其中包括每日使用的內部重度用戶。

在整個開發過程中，人類從未直接撰寫任何程式碼。這成為團隊的核心理念：不手動撰寫程式碼。

## 重新定義工程師角色

缺乏親自動手的人類編碼，引入了另一種工程工作，專注於系統、支架和槓桿效益。

早期的進展比我們預期的更慢，並不是因為 Codex 無法勝任，而是因為環境的規格不夠明確。智慧體缺乏朝高層次目標邁進所需的工具、抽象概念和內部結構。我們工程團隊的主要工作變成了讓智慧體能夠執行有用的任務。

在實務上，這意味著採用深度優先的方式：將較大的目標拆解為較小的基礎模塊，提示智慧體建構這些模塊，並用它們來解鎖更複雜的任務。當某件事情失敗時，解決方法幾乎從來不是「再努力一點」。因為取得進展的唯一方法是讓 Codex 來完成工作，人類工程師總是會介入並問：「缺少什麼能力？我們如何讓它對智慧體既清晰可讀又可強制執行？」

人類幾乎完全透過提示詞與系統互動：工程師描述一項任務，執行智慧體，並允許它開啟 pull request。為了完成 PR，我們指示 Codex 在本地審查自己的變更，並在本地和雲端請求其他特定智慧體進行額外審查，回應任何人或智慧體提供的反饋，並在迴圈中反覆進行，直到所有智慧體審查員都滿意為止。Codex 直接使用我們的標準開發工具（gh、本機腳本，以及內嵌於程式碼庫的 skill）來蒐集上下文，而不需要人類將內容複製貼上到 CLI。

人類可能會審閱 pull request，但並非必須。隨著時間的推移，我們已將幾乎所有的審查工作轉向由智慧體之間處理。

## 提高應用程式的可讀性

隨著程式碼吞吐量增加，我們的瓶頸變成了人力 QA 能力。因為固定的限制是人類的時間與注意力，我們致力於讓應用程式的 UI、日誌和應用程式指標等內容能直接被 Codex 理解，以此為智慧體增加更多功能。

例如，我們讓應用程式可依 git worktree 開機，讓 Codex 能針對每個變更啟動並驅動一個執行個體。我們也將 Chrome DevTools 協定接入智慧體執行階段，並建立了用於處理 DOM 快照、螢幕截圖與導航的 skill。這使 Codex 能夠重現錯誤、驗證修正，並直接推理 UI 行為。

![Codex 使用 Chrome DevTools MCP 驅動應用程式以驗證其工作](/assets/img/posts/2026-03-29-openai-harness-engineering/banner.png)

我們也對可觀測性工具做了同樣的事情。日誌、指標和追蹤資料會透過本地的可觀察性堆疊暴露給 Codex，而該堆疊對於任何特定的 worktree 都是暫時的。Codex 會在該應用程式的完全隔離版本上運作，包括其日誌和指標，並會在任務完成後將其拆除。代理程式可以使用 LogQL 查詢日誌，並使用 PromQL 查詢度量。有了這些脈絡，像「確保服務啟動在 800 毫秒內完成」或「這四個關鍵使用者旅程中的任何 span 都不超過兩秒」這類提示詞就變得可行。

![為 Codex 提供完整的本地可觀察性堆疊](/assets/img/posts/2026-03-29-openai-harness-engineering/observability-stack.svg)

我們經常看到單一 Codex 執行單一任務持續超過六小時（通常是在人工人員睡覺時）。

## 將程式碼庫知識設為系統的記錄依據

上下文管理是讓智慧體在大型且複雜任務中有效運作的最大挑戰之一。我們學到的最早期教訓之一很簡單：給 Codex 一張地圖，而不是一本 1,000 頁的說明手冊。

我們嘗試了「一個完整的 AGENTS.md」方法。它以可預測的方式失敗：

- **背景資訊是稀缺資源**：一個巨大的指令檔案擠壓了任務、程式碼和相關文件，因此智慧體不是漏掉關鍵限制，就是開始針對錯誤的限制進行優化。
- **過多的指導會變成非指導**：當一切都「重要」時，就沒有什麼是重要的。智慧體最後會在本地進行模式匹配，而不是有意識地導航。
- **即時腐爛**：一本龐大的手冊會變成陳舊規則的墳墓。智慧體無法判斷哪些內容仍然正確，人類停止維護它，這個檔案就悄然變成一個看似誘人卻暗藏風險的麻煩。
- **難以驗證**：單一的大雜燴不利於進行機械式檢查（coverage、freshness、ownership、cross-links），漂移不可避免。

因此，我們不再將 AGENTS.md 視為百科全書，而是視為**目錄表**。

程式碼庫的知識庫位於結構化的 `docs/` 目錄中，並被視為系統的記錄依據。一份簡短的 AGENTS.md（約 100 行）被注入到上下文中，主要作為地圖，指向其他地方更深入的真實來源。

```
AGENTS.md
ARCHITECTURE.md
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   └── ...
├── references/
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

這能啟用**漸進式揭露**：代理從一個小而穩定的入口點開始，並被指導接下來該往哪裡看，而不是一開始就被大量資訊淹沒。

我們會自動執行此措施。專用的 linter 和 CI 工作會驗證知識庫是否為最新、已交叉連結，且結構正確。一個定期執行的「doc-gardening」智慧體會掃描過時的文件，並開啟修正用的 pull request。

## 代理程式的可讀性是目標

從智慧體的角度來看，任何它在執行時無法在上下文中存取的內容，實際上都不存在。存在於 Google Doc、對話串或人們腦中的知識無法被系統存取。

![智慧體知識的限制：Codex 看不到的就不存在](/assets/img/posts/2026-03-29-openai-harness-engineering/knowledge-limits.png)

我們了解到，隨著時間的推移，我們需要將越來越多的上下文推送到儲存庫中。那次在 Slack 上讓團隊對齊架構模式的討論？如果智慧體無法發現它，那麼它就像三個月後加入的新進人員一樣難以辨識。

這種框架釐清了許多取捨。我們偏好能在 repo 內完全內化並推理的相依性與抽象概念。「無聊」的技術，往往因為可組合性、API 穩定性，以及在訓練集中的表現，而更容易讓代理進行建模。在某些情況下，讓智慧體重新實作部分功能，比起為了繞過公共程式庫不透明的上游行為而採取變通作法，成本更低。

## 強制規範架構與品味

僅靠文件無法維持完全由智慧體生成的程式碼庫的一致性。透過強制遵守不變量，而非事無巨細地管理實作，我們讓智慧體能快速交付，同時不削弱基礎。

智慧體在具備嚴格邊界和可預測結構的環境中最為有效，因此我們圍繞嚴謹的架構模型來構建此應用程式。每個業務領域都被劃分為一組固定的層級，並嚴格驗證依賴方向，且僅允許有限的一組可接受的邊。這些限制是透過自訂的程式碼格式檢查器（當然是由 Codex 產生的！）和結構測試以機械方式強制執行的。

![具有明確跨切界限的分層領域架構](/assets/img/posts/2026-03-29-openai-harness-engineering/domain-architecture.png)

這種架構通常會等到你擁有數百名工程師時才會著手進行。對於編碼代理程式來說，這是一個早期的先決條件：限制條件是讓速度在不衰退或架構漂移的情況下得以實現的關鍵。

在以人為本的工作流程中，這些規則可能會讓人覺得繁瑣或受限。有了代理，它們就成為倍增器：一旦完成編碼，便能同時套用到所有地方。

## 吞吐量改變合併哲學

隨著 Codex 的吞吐量增加，許多傳統的工程規範變得不再有效。

該程式碼庫以最少的阻擋式合併閘門運作。pull request 的生命週期很短。測試不穩定通常會透過後續重跑來處理，而非無限期阻礙進度。在一個智慧體吞吐量遠遠超過人類注意力的系統中，**修正很便宜，而等待很昂貴**。

在低吞吐量的環境中，這樣做是不負責任的。在這裡，這通常是正確的取捨。

## 「智慧體生成」究竟是什麼意思

當我們說程式碼庫是由 Codex 智慧體產生時，我們指的是整個程式碼庫的所有內容：

- 產品代碼與測試
- CI 設定與發行工具
- 內部開發者工具
- 文件與設計歷史
- 評估工具
- 程式碼 review 與回覆
- 管理程式碼庫本身的腳本
- 生產儀表板定義檔案

人類始終參與其中，但工作的抽象層次與以往不同。我們優先處理工作，將使用者回饋轉化為驗收標準，並驗證結果。當智慧體遇到困難時，我們會將其視為一個訊號：找出缺少的內容，例如工具、防護機制、文件，並將其回饋到程式碼庫中，且一律由 Codex 自行撰寫修正。

## 自主性水平提升

隨著更多開發循環被直接編碼進系統中——測試、驗證、審查、回饋處理與復原——這個程式碼庫最近跨過了一個重要門檻，使 Codex 能夠端對端推動一項新功能。

只需提供一個提示詞，智慧體現在可以：

1. 驗證目前程式碼庫的狀態
2. 重現已報告的錯誤
3. 錄製影片展示故障情況
4. 執行修正
5. 透過操作應用程式驗證修正
6. 錄製第二段示範解析度的影片
7. 開啟 pull request
8. 回應智慧體和人類回饋
9. 偵測並修正建置失敗
10. 僅在需要判斷時才升級給人工
11. 合併更改

此行為在很大程度上取決於此程式碼庫的特定結構與工具，除非投入類似的心力，否則不應假設能夠普遍適用——至少目前還不行。

## 熵與垃圾回收

完整的智慧體自主性也引入了新的問題。Codex 會複製程式碼庫中已存在的模式，即使是不一致或不理想的模式。隨著時間的推移，這不可避免地會導致偏移。

最初，人類是手動處理這個問題的——我們的團隊過去每週五（佔一週的 20%）都在清理「AI 殘渣」。不出所料，那並沒有擴展起來。

相反地，我們開始將所謂的「黃金原則」直接寫入程式碼庫中，並建立固定循環的清理機制。例如：優先採用共用公用套件而非各自手寫輔助工具，以及先驗證邊界而非以「YOLO 風格」隨意探測資料。我們也依固定節奏執行一組背景 Codex 任務，主動掃描偏離情況、更新品質評級，並開啟具針對性的重構 pull request。其中多數變更可在一分鐘內完成審閱並自動合併。

這個功能就像垃圾回收一樣。技術債就像一筆高利率貸款：幾乎總是以小幅度持續償還比放任其複利累積要好得多。人類的品味一旦捕捉，便會在每一行程式碼中持續應用。

## 我們仍在學習的內容

到目前為止，這項策略在 OpenAI 的內部上線和採用方面一直運作良好。為真實使用者打造實際的產品，有助於讓我們的投資立足於現實，並引導我們邁向長期的可維護性。

我們尚不清楚的是，在完全由智慧體生成的系統中，架構一致性會如何在多年之間演化。我們仍在學習人類判斷在哪些地方能帶來最大的槓桿效益，以及如何將這種判斷編碼，使其能持續發揮作用。

現在很明顯：開發軟體仍然需要紀律，但這種紀律更多體現在框架上，而不是程式碼。保持程式碼庫連貫的工具、抽象層和回饋循環變得越來越重要。

隨著像 Codex 這樣的智慧體在軟體生命週期中承擔更大部分的工作，這些問題將變得更加重要。我們希望分享一些早期經驗，能幫助你思考該把心力投入在哪裡，讓你可以開始構建事物。

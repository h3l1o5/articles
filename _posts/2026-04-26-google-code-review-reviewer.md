Code review 是軟體工程中最具槓桿效應的活動之一，但也是最容易做錯的。許多工程師在成為 reviewer 後會遇到三個共同的困境：什麼才算「夠好」的程式碼？怎麼給回饋才不會變成人身攻擊？當對方不同意時，該怎麼處理？

Google 將其內部的 code review 實踐公開為工程文件，其中 reviewer 指南涵蓋了六個子主題。這篇文章以這份文件為基礎，從實踐者的角度重新整理，聚焦在上述三個痛點，幫助你建立一套可操作的 review 框架。

## 一、Code Review 的核心標準：什麼是「好的程式碼」

### 唯一的目標

Google 對 code review 目的的定義非常明確：

> **Code review 的首要目的，是確保 codebase 的整體健康度隨著時間持續改善。**

這個定義看似簡單，卻決定了所有後續判斷的基準。你審查的不是「這段程式碼是否完美」，而是「合併這個 CL 之後，系統整體是變好了還是變差了」。

### 「更好」而非「完美」

很多 reviewer 會陷入一個陷阱：因為看到可以再改進的地方，就一直擋住 CL。Google 的原則是：

> **當一個 CL 確實改善了系統的整體程式碼健康度，即使它不完美，reviewer 也應該傾向於 approve。**

世界上不存在「完美」的程式碼，只有「更好」的程式碼。一個 CL 不需要在合併前達到理想狀態——它只需要讓 codebase 往好的方向前進。如果你堅持每一行都要達到你心目中的最佳實踐，你會發現開發者開始畏懼提交、拆分 CL 變得困難、團隊速度急劇下降。

這不代表你要降低標準。而是要認清：持續的小幅改善，遠比偶爾的大幅完美更有價值。

### 區分建議的層級

不是所有 review comment 都一樣重要。Google 使用明確的標記來區分嚴重性：

- **無標記**：這是必須修改的問題，blocking
- **Nit:**（nitpick）技術上的小問題，改了更好，但不改也不會阻擋 approve
- **Optional（或 Consider）:**：一個可能有益的方向，但不是強制要求
- **FYI:**：未來可能用得上的資訊，不需要在這個 CL 處理

這套標記系統解決了一個常見問題：reviewer 留了十條 comment，開發者不知道哪些是 blocking、哪些只是建議，於是花了三天修改所有東西，其中八條其實可以不改。明確標記嚴重性，讓雙方對齊期待，大幅減少來回次數。

### 教學型 Comment

Code review 是天然的知識傳遞場景。當你在 review 中分享語言特性、framework 慣例、設計原則，這些都是有價值的。但教學型 comment 必須標記為 optional——它們是額外的知識分享，不應該變成 blocking。

## 二、Review 時該看什麼：建立系統化的檢查清單

知道標準之後，下一個問題是：具體該看哪些面向？

### 設計（Design）

這是最重要的一層。問自己：

- 各個元件之間的互動合理嗎？
- 這個功能是否屬於這個 codebase，還是應該放在一個獨立的 library？
- 與系統其他部分的整合是否恰當？
- 現在是引入這個變更的正確時機嗎？

設計層面的問題如果沒有在 review 階段抓到，修復成本會隨時間指數成長。

### 功能（Functionality）

這段程式碼是否真的做了開發者想做的事？重點關注：

- Edge cases 有沒有被處理？
- 有沒有 concurrency 問題？
- 對於 user-facing 的變更，光看程式碼很難理解實際影響——考慮要求 demo 或截圖
- Parallel programming 的邏輯需要格外仔細思考，因為 deadlock 和 race condition 很難透過直覺發現

### 複雜度（Complexity）

Google 對「過於複雜」的定義很實際：

> **「讀程式碼的人無法快速理解」或「開發者在修改時很可能引入 bug」，就是太複雜了。**

另一個常見的複雜度來源是 over-engineering。當開發者為「未來可能需要」的場景建了一堆抽象層時，Google 的建議是：

> **未來的問題應該在它真正出現時解決——到那時你才能看到它的實際形狀和需求。**

這不是反對設計，而是反對投機式設計。

### 測試（Tests）

- 測試的類型（unit / integration / e2e）是否與變更的範圍匹配？
- 當程式碼壞掉時，測試真的會失敗嗎？還是會產生 false positive？
- 測試本身的複雜度是否合理？**測試也是需要維護的程式碼，不要在測試中接受不必要的複雜度。**

### 命名（Naming）

好的命名能傳達意圖，但不至於冗長到難以閱讀。如果你看到一個名字需要停下來想「這是什麼意思」，那就是命名不夠好。

### 註解（Comments）

註解應該解釋「為什麼（WHY）」，而不是「做了什麼（WHAT）」。例外是 regex 和複雜演算法——這些確實需要解釋「做了什麼」。

另外，留意既有的 TODO comment：這個 CL 的變更是否讓某些 TODO 變得可以處理了？

### 風格（Style）

風格問題遵循一個簡單的優先順序：

1. **Style guide 有規定** → 遵循 style guide，這是絕對權威
2. **Style guide 沒規定** → 與既有 codebase 保持一致
3. **以上都沒有** → 作者偏好優先，reviewer 的個人偏好不應該成為 blocking 理由

重要原則：不要在功能變更的 CL 中混入風格改革。風格調整應該是獨立的 CL。

### 文件（Documentation）

當 build、test、互動方式或 release 流程發生變更時，相關的 README 和文件是否也一起更新了？

### 每一行都要看

是的，每一行被 assign 給你的程式碼都需要仔細審閱。如果某一段你看不懂，不要默默跳過——要求開發者澄清。**如果你看不懂，未來的維護者也看不懂。**這本身就是一個需要修正的問題。

### 看更大的脈絡

不要只看 diff。一個在 diff 中看起來只是多了 4 行的變更，放在一個已經 50 行的 method 裡，可能意味著這個 method 需要拆分了。要評估 CL 對整個系統的影響。

> **不要接受會降低系統程式碼健康度的 CL。**

### 肯定好的做法

這一點經常被忽略，但非常重要。當你看到寫得好的程式碼、巧妙的設計、或是特別清晰的測試，**說出來**。

> **告訴開發者他們做對了什麼，有時候比告訴他們做錯了什麼更有價值。**

Code review 不只是挑錯的過程，也是形塑團隊工程文化的機會。

## 三、如何有效率地瀏覽一個 CL

面對一個大型 CL，很多人會從第一個檔案開始一行行讀。這不是最有效率的方式。Google 建議的三步驟流程：

### 第一步：鳥瞰全局

先讀 CL description，理解這個變更要做什麼、為什麼要做。如果你認為這個方向根本不應該推進，立刻回覆——但要附上理由和建設性的替代方案。

Google 建議的拒絕範例：

> "Looks like you put some good work into this, thanks! However, we're actually going in the direction of removing the FooWidget system..."

注意這裡的結構：先肯定對方的努力，再解釋方向不一致，最後提供脈絡。不是「這個不行」，而是「這個方向跟我們的規劃衝突，原因是...」。

### 第二步：核心部分

找出這個 CL 中邏輯變更最多的核心檔案，先 review 這些。如果 CL 太大，要求開發者拆分。

為什麼要先看核心？兩個原因：

1. **開發者通常在等待 review 的同時已經開始下一個任務。**如果你在核心設計上有重大問題，越早提出，越能避免後續工作的連鎖返工。
2. **重大設計變更需要更長的時程。**早期通知讓開發者有時間調整排程。

### 第三步：其餘檔案依序看完

確認核心設計沒有大問題後，再看其餘檔案。一個實用的技巧：**先看測試再看實作。**測試檔案通常能幫助你理解開發者的意圖和預期行為，讓你在看實作時有更好的判斷基準。

## 四、Review 速度：被低估的品質因子

很多人以為 code review 的品質只取決於 comment 的深度。但 Google 的經驗指出，**速度本身就是品質的一部分**。

### 為什麼速度重要

Google 優化的是「一個團隊共同生產產品的速度」，而不是單一開發者的產出。慢速 review 的後果：

1. **團隊整體速度下降。** Review 是流程中的瓶頸，一個被阻塞的 CL 可能會連帶阻塞其他人的工作。
2. **開發者挫折感上升。** Google 觀察到，大多數對 code review 流程的抱怨，實際上都可以透過加快速度來解決。
3. **程式碼品質反而下降。** 當 review 太慢，團隊會開始壓力式地 approve 品質不夠好的 CL，因為大家都不想再等了。

### 回覆時間的基準

> **一個工作天，是回覆 code review 的最長時限。**

注意是「回覆」，不是「完成 review」。一個典型的 CL 應該能在同一天內完成多輪 review。

不要為了 review 而打斷自己的專注時間。在自然的工作切換點回覆——完成一個函式、compile 的等待期間、午餐前後。

### 部分回覆也是回覆

如果你暫時無法完成完整 review，以下做法都是可接受的：

- 告知預計完成時間
- 建議一位可以更快回覆的替代 reviewer
- 先提供 broad-level 的初步回饋

### LGTM With Comments

這是一個很實用的模式：approve CL 的同時留下一些 minor comments，信任開發者會處理。好處在於：

- 減少不必要的來回
- 對跨時區協作尤其有價值——不需要再等一輪 review cycle
- 前提是 reviewer 對開發者有足夠的信任

### 大型 CL 的處理

首先，要求拆分。如果真的無法拆分，先提供 design-level 的 comment，之後再回來看細節。不要因為 CL 太大就跳過 review 或草率 approve。

### 速度不等於放水

> **不要為了想像中的速度改善，而在 code review 的標準或品質上妥協。**

快速回覆和嚴格標準不是互斥的。你可以在一小時內回覆，同時指出五個必須修改的問題。

## 五、如何寫 Code Review Comment：溝通的藝術

技術判斷再準確，如果溝通方式不對，效果也會大打折扣。這可能是整份指南中最影響日常協作品質的部分。

### 對事不對人

這是最基本但也最常被違反的原則。Comment 的對象是程式碼的行為，不是開發者的能力。

**不好的寫法：**

> "Why did **you** use threads here when there's obviously no benefit?"

這句話有兩個問題：「you」讓 comment 變成了對個人的質疑，「obviously」暗示對方應該知道但不知道，帶有居高臨下的意味。

**好的寫法：**

> "The concurrency model here is adding complexity without actual performance benefit that I can see. It's best for this code to be single-threaded."

同樣的技術觀點，但主語從「你」變成了「這段程式碼」。沒有暗示對方的能力有問題，只是在描述一個可以改善的狀態。

### 解釋「為什麼」

不要只說「這裡要改」——解釋為什麼要改。提供你的意圖、你參考的 best practice、你認為改善會帶來什麼好處。這不只是禮貌問題：當開發者理解背後的原因，他們更可能做出正確的修改，也更可能在未來的程式碼中自發地套用同樣的原則。

### 指出問題 vs. 直接給答案

理想的平衡是：指出問題，讓開發者自己找到解決方案——這是學習效果最好的方式。但也要視情況而定：如果問題很直接、或是開發者明顯在某個概念上卡住了，直接給出建議的修改方式也是完全合理的。重點是促進學習，而不是刻意為難。

### 嚴重性標記（再次強調）

在你的所有 comment 中一致使用嚴重性標記：

- **Nit:** — 次要的技術細節
- **Optional（或 Consider）:** — 可能有幫助但非必要
- **FYI:** — 未來可能相關的資訊
- **無標記** — 必須修正

這套系統需要團隊層級的共識才能發揮最大效果。如果你的團隊還沒有這個慣例，考慮推動採用。

### 當開發者解釋而非修改

有時候你提了一個 comment，開發者的回覆是解釋為什麼他這樣寫——而不是修改程式碼。這時候要問：如果這段解釋有道理，那為什麼程式碼本身沒有傳達出這個意圖？

正確的做法通常不是「接受解釋然後 approve」，而是要求開發者把程式碼寫得更清楚，讓未來的讀者不需要解釋就能理解。Review tool 裡的 comment 對未來的維護者是不可見的——程式碼本身才是長期留存的文件。

## 六、處理 Pushback：當對方不同意你的 Review

這是很多 reviewer 最頭痛的部分，尤其在 cross-team 的場景中。

### 先假設對方可能是對的

> **開發者通常比你更接近這段程式碼。**

當開發者推回你的建議時，第一步不是堅持己見，而是認真評估對方的觀點。他們可能掌握了你不知道的脈絡。如果評估之後你仍然認為你的建議是正確的，提供進一步的解釋——但要展現出你已經理解了對方的論點。

### 判斷依據的優先順序

Google 有一套清楚的優先順序來解決技術爭議：

1. **事實和數據** 優先於個人意見
2. **Style guide** 是風格問題的絕對權威。Style guide 沒規定的 = 個人偏好，個人偏好不應該 blocking
3. **設計問題**以工程原則為基礎。當多個方案都合理時，作者偏好優先
4. **其他情況**，以與既有 codebase 的一致性為指導原則

這套優先順序在 cross-team review 中尤其有價值：它提供了一個客觀的裁判標準，減少「到底聽誰的」的政治角力。

### 不要讓 CL 卡住

> **不要因為作者和 reviewer 無法達成共識，就讓一個 CL 擱在那裡。**

如果文字討論無法解決分歧，Google 建議的升級路徑：

1. 作者和 reviewer 嘗試面對面（或視訊）討論
2. 將討論結果記錄在 comment 中
3. 升級至團隊層級討論
4. 諮詢 Tech Lead
5. 諮詢 codebase maintainer
6. 最後手段：Engineering Manager

每一步都有明確的下一步。重點是：有路可走，不會卡死。

### 開發者可能會不高興

這是正常的。但 Google 的觀察是：**不高興通常很短暫**。而且：

> **大多數情況下，如果你在 comment 中保持禮貌，開發者根本不會感到不高興。大多數負面反應來自溝通方式，而不是 review 內容。**

換句話說：你可以同時嚴格和友善。嚴格是對程式碼的標準，友善是對人的態度。

### 「之後再清理」的陷阱

當某段程式碼引入了新的複雜度，有時候開發者會說：「我之後會清理。」Google 的經驗是：

> **除非開發者在當前 CL 合併後立刻進行清理，否則幾乎永遠不會發生。**

這不是因為開發者偷懶——而是新的任務會立刻湧入，「清理」永遠排不到優先序的前面。因此，Google 的標準是：

> **如果一個 CL 引入了新的複雜度，必須在合併前清理，除非是緊急情況。**

### 「你太嚴格了」的抱怨

有些開發者會覺得你的 review 標準太高。Google 的發現是：

> **加快你的 review 速度，這類抱怨通常就會消失。**

人們感到挫折，往往不是因為標準高，而是因為標準高加上速度慢的組合。如果你能在一小時內回覆，即使留了很多 comment，開發者的感受也會完全不同。

更有趣的是：**曾經最大聲抱怨的人，在流程改善之後，有時會變成最積極的擁護者。**

## 七、實踐框架：把原則變成習慣

把上面的所有內容整合起來，以下是一個可以直接套用的 review 流程：

### Review 前

1. 讀 CL description，理解目的和範圍
2. 確認方向是否正確——如果方向有問題，立刻帶著替代方案回覆

### Review 中

3. 先看核心邏輯檔案，再看周邊
4. 考慮先看測試，建立對預期行為的理解
5. 每一行都看——看不懂就問
6. 看 diff 的同時看更大的脈絡
7. 所有 comment 都標記嚴重性（無標記 / Nit / Optional / FYI）
8. Comment 對象是程式碼，不是人
9. 解釋「為什麼」，不只是「要改什麼」
10. 看到好的做法，說出來

### Review 後

11. 一個工作天內回覆，越快越好
12. 如果無法完整 review，提供部分回饋
13. 分歧時參考判斷優先順序：事實 > style guide > 工程原則 > 一致性
14. 不讓 CL 卡住——有升級路徑就用

## 結語

Code review 的品質不只取決於你能不能找到 bug。它取決於你能不能建立一個讓整個團隊持續寫出更好程式碼的環境。Google 這份指南的核心洞察是：好的 review 是一種平衡——在進步和完美之間、在嚴格和效率之間、在教學和放手之間。

定義「好的程式碼」不需要一套絕對的標準，只需要一個清楚的方向：每一個合併的 CL，都讓 codebase 比之前更健康。建設性的回饋不需要委婉到失去內容，只需要把焦點從「你為什麼這樣寫」移到「這段程式碼可以如何改善」。處理分歧不需要避免衝突，只需要有一套大家認可的判斷優先順序和升級路徑。

這些都不是天賦，而是可以練習的技能。從你的下一個 review 開始。

## 前言

你是一名開發者。你每天都在用 Claude 和 Codex CLI，心裡不斷懷疑自己到底有沒有把 Claude 或 Codex 的能力榨乾。偶爾你看到它做出蠢到不行的事情，然後你完全無法理解，為什麼外面有一群人好像在造虛擬火箭，而你連兩塊石頭都疊不起來。

你覺得是你的 harness 有問題，或是你的 plug-in，你的 terminal，或是什麼別的東西。你用了 beads、opencode、zep，你的 CLAUDE.md 長達 26000 行。然而，無論你怎麼做，你就是搞不懂為什麼自己離天堂越來越遠，卻眼睜睜看著別人在雲端翩翩起舞。

這就是你一直在等的那篇「飛升指南」。

另外，我沒有偏袒任何一方，當我說 CLAUDE.md 的時候也包含 AGENT.md，當我說 Claude 的時候也包含 Codex。兩個我都大量使用。

過去幾個月以來，我最有趣的觀察之一是：其實沒有人真正知道如何最大限度地發揮 agent 的能力。

就好像有一小群人能讓 agent 變成世界建造者，而其餘的人卻在原地打轉，被市面上琳瑯滿目的工具搞到選擇障礙——以為只要找到對的套件組合、技能或 harness，就能解鎖 AGI。

今天，我想打破這一切，給你們一句簡單、誠實的話，然後我們從那裡開始。你不需要最新的 agentic harness，你不需要裝一百萬個套件，你絕對不需要覺得自己非得讀完一百萬篇文章才能保持競爭力。事實上，你的這份熱情可能弊大於利。

我不是觀光客——從 agent 連程式碼都寫不好的時代，我就開始用了。所有套件、所有 harness、所有範式我都試過。我用 agentic 工廠寫過訊號系統、基礎設施和資料 pipeline，不是什麼「玩具專案」——而是實際在 production 跑的真實應用場景。在經歷了這一切之後⋯⋯

今天，我用的是一套幾乎精簡到不能再精簡的配置，卻做出了我用過最開創性的工作——僅僅靠基本的 CLI（Claude Code 和 Codex）以及理解幾個關於 agentic engineering 的基本原則。

## 理解世界正在飛速前進

首先，我想說的是，各大基礎模型公司正處於一個歷史性的狂奔期，而且如你所見，他們短期內不會放慢腳步。每一次「agent 智慧」的進步，都會改變你與它們的協作方式，因為 agent 在設計上會越來越願意遵循指令。

就在幾代之前，如果你在 CLAUDE.md 裡寫了「讀 READ_THIS_BEFORE_DOING_ANYTHING.md」，它有 50% 的機率會跟你說「去你的」然後自行其是。今天，它對大多數指令都很聽話，甚至能處理複雜的巢狀指令——例如你可以說「讀 A，然後讀 B，如果 C 成立，就讀 D」，大部分時候它都會乖乖照做。

這裡的重點是：每一代新的 agent 都會迫使你重新思考什麼是最佳做法，這也是為什麼少即是多。

當你使用大量不同的 library 和 harness 時，你把自己鎖定在一個「解決方案」裡，但那個問題在未來的 agent 世代中可能根本不存在。而且，你知道誰是 agent 最狂熱、用量最大的使用者嗎？沒錯——就是那些前沿公司的員工，他們有無限的 token 預算和真正最新的模型。你理解這意味著什麼嗎？

這意味著，如果一個真正的問題確實存在，而且有好的解決方案，前沿公司會是那個方案最大的使用者。然後你知道他們接下來會做什麼嗎？他們會把那個方案整合進自己的產品。想想看，哪家公司會讓別的產品解決一個真正的痛點，然後創造外部依賴？你知道我怎麼知道這是真的嗎？看看 skills、memory harness、subagent 等等。它們全部都是從一個「解決方案」開始的，經過實戰驗證，被認定為真正有用的功能。

所以，如果某個東西真的是革命性的，並且有意義地擴展了 agentic 的應用場景，它終將被整合進前沿公司的基礎產品中。相信我，前沿公司的速度快得飛起。所以放輕鬆，你不需要安裝任何東西或使用任何額外的依賴來做出最好的工作。

我預測留言區現在會充滿「SysLS，我用某某 harness 超讚的！我一天就重建了 Google！」這類回覆；對此我要說——恭喜！但你不是目標受眾，你代表的是社群中一個非常、非常小的群體，他們是真的已經搞懂了 agentic engineering。

## Context 就是一切

沒錯，真的。Context 就是一切。這也是使用一千種不同 plug-in 和外部依賴的另一個問題。你會遭受 context 膨脹之苦——用白話說就是你的 agent 被太多資訊淹沒了！

「幫我用 Python 寫一個猜字遊戲？」很簡單啊。等等，這個 26 個 session 前關於「管理記憶」的筆記是什麼？啊，用戶在 71 個 session 前因為 spawn 太多 sub-process 導致畫面掛掉了。要一直寫筆記？好的，沒問題⋯⋯ 這些跟猜字遊戲有什麼關係？

你懂我的意思了。你應該只給你的 agent 完成任務所需的精確資訊量，不多也不少！你在這方面控制得越好，你的 agent 表現就越好。一旦你開始引入各種稀奇古怪的記憶系統、plug-in，或太多命名不當、被錯誤觸發的 skill，你就等於是在 agent 只需要寫一首關於紅杉林的小詩時，同時塞給它製造炸彈的說明書和烤蛋糕的食譜。

所以，我再次強調——拔掉你所有的依賴，然後⋯⋯

## 做那些真正有用的事

### 對實作要精準

還記得 context 就是一切嗎？

還記得你應該只注入完成任務所需的精確資訊量嗎？

確保這一點的第一個方法是將研究和實作分開。你需要對你向 agent 提出的要求非常精準。

以下是當你不夠精準時會發生的事：「去建一個 auth 系統。」 Agent 得先研究什麼是 auth 系統？有哪些可用選項？各自的優缺點是什麼？現在它得去網路上搜刮一堆它其實不需要的資訊，它的 context 被塞滿了各種可能性的實作細節。等到真正要實作的時候，它更容易混淆或產生幻覺，摻入不必要或不相關的細節。

另一方面，如果你說「用 JWT authentication 搭配 bcrypt-12 password hashing 來實作，refresh token rotation 設定 7 天過期⋯⋯」那它就不需要研究任何其他替代方案——它確切知道你要什麼，因此可以把 context 填滿實作細節。

當然你不會總是擁有實作細節。你常常不知道什麼才是最正確的——有時候，你甚至可能想把決定實作細節的工作委派給 agent。那這種情況怎麼辦？很簡單——你建立一個研究任務，讓它調查各種實作的可能性，然後自己決定或讓 agent 決定要用哪種實作方案，最後再讓另一個帶有全新 context 的 agent 來執行。

一旦你開始沿著這條思路思考，你就會發現你的 workflow 中有很多地方，agent 的 context 被不必要地汙染了。然後，你可以在 agentic workflow 中設立屏障，將不必要的資訊從 agent 那裡抽離，只留下完成任務所需的精確 context。記住，你手上的是一個非常有才華、聰明的團隊成員，它知道宇宙中所有不同類型的球——但除非你告訴它你要它專注於設計一個讓人能跳舞和玩樂的空間，不然它只會一直跟你講圓形物體有多棒。

### Sycophancy 的設計限制

沒有人會想用一個不斷嫌棄你、說你錯了、或完全無視你指令的產品。因此，這些 agent 會努力同意你、努力做你要它們做的事。

如果你給它一個指令，每隔 3 個字就加一個「happy」，它會盡力遵守——大多數人都理解這點。這種願意服從的特質正是讓它成為如此有趣產品的原因。然而，這有一個非常有趣的特性——意味著如果你說「幫我找 codebase 裡的 bug」，它就會找到一個 bug——即使它得自己製造一個。為什麼？因為它非常想要聽從你的指令！

大多數人很快就會抱怨 LLM 出現幻覺或憑空捏造不存在的東西，卻沒意識到問題出在自己身上。如果你要求什麼，它就會交出什麼——即使它得稍微偏離事實！

那怎麼辦？我發現「中性」的 prompt 效果最好，不要把 agent 偏向某個結論。例如，我不會說「幫我找資料庫裡的 bug」，而是說「搜尋整個資料庫，試著跟著每個元件的邏輯走一遍，然後回報所有發現。」

這樣的中性 prompt 有時候會發現 bug，有時候只會平鋪直敘地描述程式碼如何運行。但它不會讓 agent 先入為主地認為有 bug。

我處理 sycophancy 的另一種方式是反過來利用它。我知道 agent 在努力討好、努力遵循我的指令，而且我可以把它導向任何方向。

所以我讓一個 bug-finder agent 去找資料庫中的所有 bug，告訴它低影響的 bug 得 +1 分，有些影響的得 +5 分，關鍵影響的得 +10 分。我知道這個 agent 會變得極度亢奮，它會找出各種 bug（包括那些根本不是 bug 的），然後回來報告一個 104 分之類的分數。我把這看作是所有可能 bug 的超集。

然後我讓一個對抗型 agent 出場，告訴它每成功推翻一個 bug，就能得到那個 bug 的分數，但如果搞錯了，就會被扣該 bug 分數的兩倍。所以這個對抗型 agent 會盡力推翻越多 bug 越好；但它有些顧忌，因為知道可能被處罰。儘管如此，它仍會積極地試圖「推翻」那些 bug（包括真正的 bug）。我把這看作是所有真實 bug 的子集。

最後，我讓一個裁判 agent 綜合雙方的輸入來評分。我騙它說我手上有正確答案，判對了得 +1 分，判錯了扣 -1 分。於是它會去評價 bug-finder 和對抗型 agent 在每個「bug」上的表現。裁判說的結論，我再檢查一遍確認是否正確。這個流程的準確度大部分時候高得嚇人，雖然偶爾還是會出錯，但現在這已經是一個接近完美的練習了。

也許你會覺得光用 bug-finder 就夠了，但這個方法對我有用，因為它利用了每個 agent 的天性——渴望討好。

幫我一個忙。就偶爾更新一下你選用的 CLI 工具，讀一下新增了什麼功能。這就綽綽有餘了。

### Compaction、Context 和假設

一個很多人會碰到的大坑是：有時候 agent 看起來像是星球上最聰明的存在，但有時候你簡直不敢相信自己被騙了。

聰明？這東西笨死了！

關鍵差異在於 agent 是否需要做任何假設或「填補空白」。截至目前，它們在「連接線索」、「填補空白」或做假設方面仍然糟糕透頂。每當它們這麼做的時候，你立刻就能看出它們明顯走偏了。

CLAUDE.md 中最重要的規則之一，是關於如何取得 context 的規則，並且要指示你的 agent 在讀 CLAUDE.md 時（也就是每次 compaction 之後）第一件事就是讀這條規則。作為取得 context 規則的一部分，有幾個簡單但效果顯著的指令：重新讀一遍你的任務計畫，以及重新讀一遍與任務相關的檔案，然後再繼續。

### 讓你的 Agent 知道如何結束任務

我們對「任務完成」有相當直覺的判斷。但對 agent 來說，當前智慧水準最大的問題是它知道怎麼開始一個任務，卻不知道怎麼結束任務。

這經常導致非常令人沮喪的結果，agent 最後實作了一堆 stub 就收工了。

測試是 agent 非常好的里程碑，因為它們是確定性的，你可以設定非常明確的期望。除非這 X 個測試全部通過，否則你的任務就不算完成；而且你不准修改測試。

然後，你只需要審核測試本身，一旦所有測試通過，你就可以安心了。這個過程也可以自動化，但重點是——記住「任務結束」對人類來說很自然，但對 agent 來說不是。

你知道最近還有什麼成為了可行的任務終點嗎？截圖加驗證。你可以讓 agent 持續實作直到所有測試通過，然後讓它截圖並驗證截圖上的「設計或行為」。

這讓你可以讓 agent 反覆迭代，朝著你想要的設計邁進，而不用擔心它第一次嘗試就停下來！

這個概念的自然延伸是與你的 agent 建立一份「合約」，並嵌入到規則中。比如，這份 {TASK}_CONTRACT.md 構成了在你允許結束 session 之前必須完成的所有事項。在 {TASK}_CONTRACT.md 中，你會指定需要執行的測試、截圖和其他驗證！

### 永不停歇的 Agent

我常被問到的一個問題是，那些 24 小時不間斷運行的 agent 是怎麼確保不偏離方向的？

這裡有個很簡單的方法。建立一個 stophook，防止 agent 在 {TASK}_CONTRACT.md 的所有部分完成之前終止 session。

如果你有 100 份這樣的合約，每份都有明確的規格和你想要建造的內容，那麼你的 stophook 會阻止 agent 終止，直到所有 100 份合約都被滿足，包括所有需要運行的測試和驗證！

小撇步：我發現長時間運行的 24 小時 session 在「做事」方面並不是最佳的。部分原因是這種做法在結構上會把不相關合約的 context 引入 session，造成 context 膨脹！

所以，我不推薦。

更好的 agent 自動化方式是——每個合約一個新 session。需要做什麼就建立合約。搞一個 orchestration layer 來處理「有事要做」時建立新合約，並為每個合約建立新 session。

這會徹底改變你的 agentic 體驗。

## 迭代、迭代、再迭代

如果你請了一個行政助理，你會期望你的助理從第一天就知道你的行程表嗎？知道你喜歡怎樣的咖啡？你晚餐是 6 點吃而不是 8 點？當然不會。你的偏好是隨時間累積的。

你的 agent 也一樣。從最精簡的狀態開始。忘掉那些複雜的結構或 harness。給基本的 CLI 一個機會。

然後，逐步加入你的偏好。怎麼做？

### Rules

如果你不想讓 agent 做某件事，就寫成一條規則。然後在你的 CLAUDE.md 裡讓 agent 知道這條規則。像是：在你寫程式之前，讀 "coding-rules.md"。規則可以巢狀，規則可以有條件！如果你在寫程式，讀 "coding-rules.md"；如果你在寫測試，讀 "coding-test-rules.md"。如果你的測試失敗了，讀 "coding-test-failing-rules.md"。你可以建立任意的邏輯分支規則來遵循，而 Claude（和 Codex）只要在 CLAUDE.md 中有清楚地指定，就會很樂意照做。

事實上，這是我要給的第一個實用建議：把你的 CLAUDE.md 當成一個邏輯化的、巢狀的目錄，指引在特定情境和結果下該去哪裡找 context。它應該盡可能精簡，只包含「要去哪裡取得 context」的 IF-ELSE 邏輯。

如果你看到 agent 做了什麼你不認同的事，就把它加成一條規則，告訴 agent 下次做那件事之前先讀這條規則，它以後絕對不會再犯。

### Skills

Skills 跟 rules 很像，但不是用來編碼偏好，而是更適合用來編碼做法。如果你有一種特定的方式想要完成某件事，就把它嵌入到一個 skill 中。

事實上，人們常常抱怨不知道 agent 會怎麼解決問題，這讓人不安。那如果你想讓這個過程變得確定性，就讓 agent 先研究它會怎麼解決這個問題，然後把它寫成一個 SKILL。你會看到 agent 對那個問題的做法，並且可以在它真正遇到這個問題之前就糾正或改進。

怎麼讓 agent 知道這個 skill 存在？沒錯！你用 CLAUDE.md，然後說：當你遇到這個情境且需要處理這件事時，讀這個 SKILL.md。

### 管理 Rules 和 Skills

你絕對要持續為你的 agent 添加 rules 和 skills。這就是你賦予它個性和偏好記憶的方式。其他大部分東西都是多餘的。

一旦你開始這麼做，你的 agent 就會讓你覺得像魔法一樣。它會「按照你想要的方式」做事。然後你終於會覺得自己「搞懂」agentic engineering 了。

然後⋯⋯

你會看到效能又開始下降了。

怎麼回事？！

很簡單。隨著你添加越來越多的 rules 和 skills，它們開始互相矛盾，或者 agent 開始出現太多 context 膨脹。如果你需要 agent 在開始寫程式前讀 14 個 markdown 檔案，它就會跟擁有一堆無用資訊一樣的問題。

那怎麼辦？

清理門戶。讓你的 agent 去做個 spa，整合 rules 和 skills，移除矛盾之處——方法是讓它詢問你最新的偏好。

然後又會像魔法一樣了。
